# AGENTS.md

This file provides guidance to AI Agents when working with code in this repository.

## Commands

Package manager is **pnpm** (version pinned in `package.json`). Node version is pinned in `.node-version`.

- `pnpm dev` ‚Äî Vite dev server on port **8080**
- `pnpm build` ‚Äî production build into `.output/`
- `pnpm preview` ‚Äî preview the production build locally
- `pnpm test` ‚Äî run Vitest (see "Testing" below for project split)
- `pnpm lint` ‚Äî ESLint (flat config based on `@tanstack/eslint-config`)
- `pnpm format` / `pnpm format:check` ‚Äî Prettier (with import-sort and Tailwind plugins)
- `pnpm typecheck` ‚Äî runs `cf-typegen` first, then `tsc --noEmit` for both app and `tests/tsconfig.json`
- `pnpm cf-typegen` ‚Äî builds, then regenerates `worker-configuration.d.ts` from `.output/server/wrangler.json` (not from `wrangler.jsonc` directly)
- `pnpm check` ‚Äî the CI gate: `format:check && lint && typecheck && deploy:dry-run`
- `pnpm deploy` / `pnpm deploy:dry-run` ‚Äî publishes via `wrangler --cwd .output`; deploy-dry-run is part of `check` to catch Worker-level misconfig before merge

Run a single test file: `pnpm test <path>` (e.g. `pnpm test tests/features/prompt-library/server/prompt-library-functions.test.ts`). Filter by name with `-t "<pattern>"`.

Apply D1 migrations locally: `pnpx wrangler d1 migrations apply promptrc --local`.

## Architecture

### Local-first storage with a unified client

The prompt library has two backends behind one interface:

- `src/features/prompt-library/storage/prompt-library-storage.ts` ‚Äî the `PromptLibraryStorage` discriminated union (`mode: 'local' | 'remote'`)
- `local-prompt-library-storage.ts` ‚Äî `localStorage` adapter (anonymous users)
- `remote-prompt-library-storage.ts` ‚Äî Cloudflare D1 via TanStack server functions (signed-in users)
- `prompt-library-client.ts` ‚Äî `createPromptLibraryClient` builds a single `PromptLibraryClient` on top of either storage; components never branch on auth/mode

`PromptLibraryProvider` (`components/prompt-library-provider.tsx`) wires Clerk auth ‚Üí `usePromptLibraryStorage` (picks adapter) ‚Üí `createPromptLibraryClient` ‚Üí Zustand store. Hydration happens in `usePromptLibraryHydration`: local mode subscribes to the store to persist every change; remote mode does not (writes go through server funcs explicitly).

**Implication:** don't add mode checks in components/hooks. If a behavior differs between local and remote, it belongs inside the client/storage layer.

### Server functions + Cloudflare bindings

`src/features/prompt-library/server/prompt-library-functions.ts` defines TanStack Start `createServerFn` endpoints. Important patterns:

- The D1 binding is accessed via `const { env } = await import('cloudflare:workers')` inside `getDatabase()` ‚Äî **not** a top-level import. Vite externalizes `cloudflare:workers` (see `vite.config.ts` `build.rollupOptions.external`), so a static import would break the client bundle.
- Every handler calls `requireUserId()` (Clerk `auth()`) and scopes queries by `ext_user_id`. New endpoints must follow this pattern; there is no separate authorization layer.
- Exported pure DB helpers (`listPromptsForUser`, `upsertPromptForUser`, etc.) exist so tests can call them with a direct `D1Database` handle without going through the serverfn wrapper.

### State

Zustand **vanilla** store (`store/prompt-library-store.ts`) ‚Äî the React tree consumes it through `useStore(store, selector)` exposed by `PromptLibraryProvider`. The store holds prompts, workspace state (query/selection/composer), and sync state together; mutations go through the `actions` slot (never direct `set` from outside).

### Routing

File-based routes in `src/routes/`. The generated `routeTree.gen.ts` is `.gitignored` in effect (regenerated by the TanStack router plugin during `dev`/`build`). `sign-in.$.tsx` / `sign-up.$.tsx` are catch-all Clerk routes.

### Feature layout convention

Colocate under `src/features/<feature>/`: `components/`, `hooks/`, `lib/` (pure helpers), `server/` (serverfn), `storage/` (adapters/persistence), `store/` (Zustand), `types.ts`. New features should follow this same shape.

### Path alias

`@/*` ‚Üí `src/*` (tsconfig + vitest alias). Use the alias in imports; Prettier's import-sort plugin groups `^@/(.*)$` separately from third-party and relative imports ‚Äî don't hand-reorder.

## Testing

`vitest.config.ts` defines **two projects** in one run:

- `browser-unit` ‚Äî `jsdom`, includes `tests/**/*.test.{ts,tsx}` but **excludes** `tests/**/server/**`
- `cloudflare-server` ‚Äî `@cloudflare/vitest-pool-workers` Miniflare, includes `tests/**/server/**`, runs `tests/setup-cloudflare.ts` which re-applies D1 migrations and truncates `prompts` before each test

**Placement matters:** server-function tests must live under a `server/` directory so they hit the workers pool (and get the D1 binding). Put anything else elsewhere, or it'll run in jsdom without a DB.

## Deployment

Target: Cloudflare Workers with a D1 binding named `DB` (see `wrangler.jsonc`). `nitro.config.ts` sets the Nitro preset to Cloudflare; retargeting elsewhere is a preset change. `pnpm build` outputs the Worker bundle + `.output/server/wrangler.json`, and `pnpm deploy` runs wrangler against that output directory ‚Äî never the repo root.

Production routes (`promptrc.app`, `www.promptrc.app`) and the D1 database id are pinned in `wrangler.jsonc`. Fork-to-deploy requires editing those and setting `CLERK_SECRET_KEY` as a Worker secret.

## Environment

- `VITE_CLERK_PUBLISHABLE_KEY` ‚Äî client, required
- `CLERK_SECRET_KEY` ‚Äî server/worker only, required
- `VITE_SITE_URL` ‚Äî optional canonical origin for SEO; defaults handled by `src/lib/site-config.ts`

Only `VITE_*` vars reach the client bundle.

## Conventions

- Prettier: **no semicolons**, single quotes, trailing commas. Don't fight it.
- ESLint: `array-simple` for array types; inline type specifiers (`import { type Foo }`) are enforced.
- Adding a shadcn component: `npx shadcn@latest add <name>` (registry config in `components.json`).
